1. Kernel 概述

1.1 功能描述

该 Kernel 在基础版本 hgemm_mma_m16n8k16_mma2x4_warp4x4_kernel 基础上增加了两项关键优化：

  1. Multi-stage Pipeline: 使用异步拷贝实现多阶段流水线，隐藏访存延迟
  2. Block Swizzle: 重新映射 Block 与矩阵位置的关系，提升 L2 Cache 命中率

5.6 stride 参数的选择

stride = 2048 的选择依据：

  1. L2 Cache 容量匹配
  • A100 L2 cache = 40 MB
  • 一条带的 B 数据量: stride × K × sizeof(half) = 2048 × 4096 × 2 = 16 MB
  • 加上 A 数据，总量仍可容纳在 L2 cache 中
  1. Block 数量平衡
  • stride / BN = 2048 / 128 = 16 blocks 为一组
  • 16 个 Block 形成一个 “复用组”，共享 A 矩阵的相同行
  1. 经验值
  • 太小：Swizzle 效果不明显
  • 太大：超出 L2 cache 容量，导致 cache miss
  • 2048 是在多种 GPU 上验证过的良好平衡点

5.7 Block Swizzle 的本质

核心问题：Block Swizzle 本质上是根据 Block 的执行顺序，通过映射函数改变 Block 计算的矩阵位置，以达到按顺序执行的 Block 能最大化数据复用。

5.8.3 为什么 stride = 2048？

stride = 2048 的选择基于以下考量：

  1. L2 Cache 大小匹配
  • A100 L2 cache = 40 MB
  • 每个 Block 处理 128×128 = 16K 元素 = 32KB (half 类型)
  • 一条带(stride=2048)内的 B 数据：2048 × K × sizeof(half)
  • 对于 K=4096：2048 × 4096 × 2 = 16MB，适合 L2 cache
  1. 与 Tile 大小的关系

意味着每 16 个 Block 形成一个"复用组"，共享 A 矩阵的同一行 Tile

  1. 实践经验值
  • 太小（如 512）：Swizzle 效果不明显，复用组太小
  • 太大（如 8192）：超出 L2 cache 容量，导致 cache miss
  • 2048 是多种 GPU 上验证过的良好平衡点

5.8.4 N_SWIZZLE 计算示例

矩阵 N stride N_SWIZZLE 每带 Block 数 说明
1024 2048 1 8 矩阵小，无需 swizzle
2048 2048 1 16 刚好一条带
4096 2048 2 16 分成 2 条带
8192 2048 4 16 分成 4 条带
16384 2048 8 16 分成 8 条带

5.8.5 如何调优 stride？

最优值取决于：

  • L2 Cache 大小：stride 对应的数据量不应超过 L2 大小
  • 矩阵尺寸：非常大的矩阵可能需要更大的 stride
  • GPU 架构：不同 GPU 的 L2 cache 大小不同
GPU L2 Cache 推荐 stride
RTX 3080 5 MB 1024~2048
A100 40 MB 2048~4096
H100 50 MB 2048~4096

总结：N_SWIZZLE = 2 并不是一个固定值，而是 ceil(N / stride) 计算得来。stride = 2048 是一个基于 L2 cache 大小和实践调优得到的经验值，确保同一 Swizzle 组内的 Block 能最大化复用 L2 cache 中的数据。

7. 性能优化要点

7.1 优化技术总结

优化技术 实现方式 效果
Multi-stage Pipeline K_STAGE=2/3/4 隐藏 Global→Shared 访存延迟
cp.async 异步拷贝指令 Load 与 Compute 重叠
Block Swizzle 重映射 Block 坐标 提升 L2 Cache 命中率
WARP_TILE_K=2 每次处理 2 个 K-tile 增加计算密度
Shared Memory Padding A_PAD=8, B_PAD=8 消除 Bank Conflict

7.2 性能提升预期

配置 vs 基础版本 说明
K_STAGE=2 +5~15% 基本流水线
K_STAGE=3 +8~20% 更深流水线
Block Swizzle +3~10% 大矩阵效果明显
组合优化 +15~30% 最佳配置

7.3 适用场景

  • 推荐使用 stages kernel：

  • 矩阵尺寸 ≥ 2048

  • K 维度较大（访存延迟明显）

  • GPU 计算能力 ≥ 8.0 (支持 cp.async)

  • 推荐启用 Block Swizzle：

  • N 维度 ≥ 2048

  • L2 cache 压力大的场景

  • 多 SM 并行度高的情况
