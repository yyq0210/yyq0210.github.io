MLA(Multi-head Latent Attention)完整详解

结合 DeepSeek-V2 / V3 论文数学公式、知乎讲解(zhuanlan.zhihu.com/p/16730036197)与 sglang 代码实现(python/sglang/srt/models/deepseek_v2.py、deepseek_common/attention_forward_methods/forward_mla.py)。

一句话:MLA 通过把 KV 压缩成一个低秩潜向量(latent)来减少 KV-cache 显存,再借助"矩阵吸收"在解码时退化成 MQA 的访存/计算模式,从而做到又省、又快、又(几乎)不掉质量。

第一部分:为什么需要 MLA —— KV-cache 是推理瓶颈

1.1 LLM 推理的两个阶段

  • Prefill(预填充):对整个 prompt 的所有 token 一次性并行计算,产出第一个输出 token。计算密集。
  • Decode(解码):逐个 token 生成(token-by-token),直到 EOS。访存密集。

1.2 KV-cache 的由来:用空间换时间

LLM 是 Causal Attention(因果注意力):位置 的 token 只与它前面的 token 交互。用一个下三角 mask 实现。

观察两个事实:

  1. 前面 token 的 不受后面 token 影响;
  2. 每生成一个新 token,都要用到所有前序 token 的 ,而这些值始终不变。

所以为避免重复计算,把前序的 缓存起来 —— 这就是 KV-cache。

代价:它把"计算密集型"任务变成了"访存密集型"任务。 不再现算,而是从显存(HBM)读出来,GPU 核心和显存之间频繁读写 → 访存速率直接决定推理速度。

1.3 显存里存了什么

推理阶段显存主要三部分:

  1. 模型参数:固定大小。
  2. KV-cache:随 (Batch × Seq_len)动态增长 —— 长上下文 / 高并发下会爆炸。
  3. 运行时中间数据:即用即释放,很小。

结论:减少 KV-cache → 单次读写更少(更快)+ 数据能塞进单卡/单机用更高带宽(更快)。这正是 MLA 的目标。

访存分级(2 机 × 8×A100):卡内带宽 单机卡间的 3 倍 跨机的 20 倍。所以数据要尽量留在卡内 → 越省显存,越能享受高带宽。

第二部分:已有的 KV-cache 优化路线

类别 思路 代表
共享 KV 多个头共享一组 KV MQA、GQA、(MLA)
窗口 KV 只保留窗口内 KV,丢弃超窗的 Longformer
量化压缩 更低 bit 存 KV INT8
计算优化 减少换入换出,多在 SRAM 算 FlashAttention

MLA 属于共享 KV 分支的变体。先看它要对比的两个基线:

2.1 MHA / MQA / GQA(单 token KV 缓存量, 层、每层 头)

方法 KV 头数 单 token 缓存量 含义
MHA 个向量 每个 query 头有专属 K/V → 最占显存,质量最好
GQA 组 个向量 每 个头共享一组 K/V → 折中
MQA 个向量 所有头共享一组 K/V → 最省显存,质量下降
  • 时 GQA 退化成 MQA; 时退化成 MHA。

核心权衡:KV 头越少越省显存越快,但越掉质量。 MLA 的野心:显存像 MQA,质量像 MHA。

第五部分:常见疑问澄清

Q1:吸收后退化成 的 MQA,为什么不掉质量? 缓存里只有 1 份共享的 (像 MQA → 省显存、访存少 → 快)。但每个 query 头点积之前先乘上自己专属的 头切片(),等于戴着各自的"眼镜"去解读同一份共享数据 → 个头看出 种不同内容 → 保住 MHA 的多头表达力。用 MQA 的代价,拿到接近 MHA 的效果。

Q2: 不也要存吗? 要,但它是权重(每层 1 份、固定、和 token 数/batch 无关),不是 KV-cache(每 token 每请求都涨)。MLA 省的是后者。 每层 固定,而 KV-cache 每层可达数 GB 且随上下文爆炸。且 只是 kv_b_proj 换形状,本就存在 → 几乎零额外开销。省显存只看"随 token 增长的部分"。

Q3:为什么非要把 RoPE 单独拆出来(nope/rope)? RoPE 是依赖相对位置的旋转矩阵 。若把它套在解压得到的 上, 会卡在 和解压矩阵 之间,而 (不可交换)→ 吸收失效,必须重算所有前缀 K。冲突的根源是 。解法:把位置信息放到一条不需要吸收的小通道(,直接投影、共享、不解压),RoPE 套在它上面无冲突。位置共享走 rope 半,语义分头走 nope 半。

第六部分:一图流总结

三句话记住 MLA:

  1. 省显存 —— 每 token/层只缓存 ,约为 MHA 的 、MQA 的 。
  2. 拆 RoPE —— 位置半单独存(,共享、不解压),因为 RoPE 与解压矩阵不可交换,会破坏吸收。
  3. 吸收 + MQA —— decode 时把 吸进 Q、 吸进输出,在压缩空间跑 的 MQA;每头各自的 保住多头表达力 → 又省、又快、又(几乎)不掉质量。
